companion manuscript submission freeze checklist

Terminal-by-terminal list to run before tagging and uploading. Every box must be ticked in the indicated order. Do NOT skip the simulation tier - it is the only check that proves a reviewer starting from a clean clone will see identical artifacts.

Scope: everything below the "Human-gated" banners is automatable; everything inside them requires human judgement, credentials, or actions that a script cannot perform safely.

1. Pre-freeze regressions

  • cd kstar-verify/lean4 && lake build exits 0
  • python scripts/check_sorry.py --layer1 reports 0 sorry
  • python scripts/generate_all.py exits 0 with "no drift"
  • python scripts/simulate_fresh_clone.py --share-cache passes ("generated/ artifacts match byte-for-byte")
  • cd kstar-verify && python verify_registry.py shows 193 passed, 0 failed (the eq:-entry checks for eq:hoeffding and eq:kappa recognise equation-type entries, allow 1 anchor, and exempt equation owners from the shared-anchor requirement, so no warnings are expected)
  • cd kstar-verify && python run_all.py --skip-tier1 --skip-lean4 all tiers pass

2. Manuscript

  • pdflatex manuscript.tex clean, no unresolved refs
  • pdflatex supplemental_material.tex clean
  • \input{generated/sm_axiom_table.tex} renders the machine-generated table on the SM page
  • Scope-of-formalization paragraph mentions Phases F and G
  • All em-dashes removed (rg -- '---' yields only code fences)

3. Data and code

  • DOI tree hashes match KV / APS copies (per three-location-sync rule: DOI is source of truth)
  • No duplicate raw data in KV / APS / auxiliary
  • .zenodo.json has the K* Verification DOI filled in (once assigned)

4. Tooling pins

  • lean4/lean-toolchain matches the version tested in CI
  • requirements.txt upper bounds still valid (no breaking major releases slipped through)

5. Repo hygiene

  • No .ipynb_checkpoints/, __pycache__/, *.executed.ipynb stragglers committed
  • README badges reference real workflow names
  • CITATION.cff preferred-citation DOI matches .zenodo.json
  • README Binder URL updated from HEAD to the submission tag (e.g., v1.0.0-submission-submission) after tag push; otherwise a future main-branch commit can break the Binder launch a reviewer clicks. Change github/kootru-repo/kstar-verify/HEAD?filepath=... to .../v1.0.0-submission-submission?filepath=... in README.md and notebook/README.md.

Human-gated: git tagging (2 repos)

Do NOT push tags yet - create locally, verify, then push after a second pair of eyes confirms.

Tag push is an irreversible public action that triggers Zenodo DOI minting. Verify the SHA of each tag against the submission manifest BEFORE pushing.

Step 1: tag kstar-verify

cd /path/to/kstar-verify
git tag -a v1.0.0-submission-submission -m "Pin for companion manuscript submission"

Step 2: tag the author's manuscript source repo (if separate)

cd /path/to/manuscript-source
git tag -a v1.0.0-submission-submission -m "Pin for companion manuscript submission"

Step 3: verify both tags agree

for repo in kstar-verify manuscript-source; do
  cd /path/to/$repo
  echo "=== $repo ==="
  git log -1 v1.0.0-submission-submission --pretty='%H %ad %s'
done

Step 4: push tags (only after ALL boxes above ticked)

git push origin v1.0.0-submission-submission    # run in each repo

Rollback

If a mistake is caught before external systems react:

git tag -d v1.0.0-submission-submission
git push origin :refs/tags/v1.0.0-submission-submission

If Zenodo has already minted a DOI, the DOI is permanent and points to that state. You can retract the record, but the DOI URL will still exist as tombstoned.

Human-gated: Zenodo DOI minting

Zenodo's GitHub integration auto-mints DOIs when a tag lands on a connected repo. To use it:

Prerequisites

  • A Zenodo account linked to the Kootru Labs GitHub organisation.
  • The kstar-verify repo enabled in Zenodo → GitHub settings.

Post-tag flow

  1. Push v1.0.0-submission-submission from the step above.
  2. Within ~5 minutes, Zenodo will pick up the tag and create a new record, using .zenodo.json as metadata.
  3. Visit Zenodo → uploads, open the new draft, and publish it.
  4. Copy the minted DOI (format: 10.5281/zenodo.NNNNNNN).
  5. Update these four places with the DOI:
    • .zenodo.json - related_identifiers[0].identifier
    • README.md - DOI badge URL
    • CITATION.cff - preferred-citation.doi
    • manuscript.tex - any citation to the verification repo
  6. Commit the DOI update and tag v1.0.1-submission-submission (DOI amendment). Zenodo will mint a second, linked DOI for this patch record.

Human-gated: GHCR Docker push

The Dockerfile.lean4 recipe builds a verification container pinned to the submission tag. Pushing it to ghcr.io/kootru-repo/kstar-verify lets reviewers pull a pre-built image without installing Lean locally.

Prerequisites

  • GHCR personal access token (PAT) with write:packages scope.
  • Docker Desktop or Docker Engine running locally.
  • About 4 GB free disk for the build layer cache.

Build and push

# Login to GHCR with your PAT.
echo $CR_PAT | docker login ghcr.io -u <github-username> --password-stdin

# Build (expect ~25 min cold; ~5 min with BuildKit cache).
cd /path/to/kstar-verify
docker build -f Dockerfile.lean4 \
  -t ghcr.io/kootru-repo/kstar-verify:submission-submission \
  -t ghcr.io/kootru-repo/kstar-verify:v1.0.0-submission-submission \
  .

# Verify the image runs and prints the manifest summary.
docker run --rm ghcr.io/kootru-repo/kstar-verify:submission-submission

# Push both tags.
docker push ghcr.io/kootru-repo/kstar-verify:submission-submission
docker push ghcr.io/kootru-repo/kstar-verify:v1.0.0-submission-submission

# Flip the package to public visibility via GitHub UI:
# https://github.com/orgs/kootru-repo/packages

Smoke test from a clean machine

docker pull ghcr.io/kootru-repo/kstar-verify:submission-submission
docker run --rm ghcr.io/kootru-repo/kstar-verify:submission-submission
# Should print: git SHA, toolchain, build exit 0, sorry=0, 28 claims summary.

Human-gated: record the 2-minute demo video

See VIDEO_DEMO_SCRIPT.html for the shot-by-shot storyboard. Recording itself is manual; the procedure:

Tools

  • OBS Studio for capture.
  • OpenShot or Kdenlive for optional post-production and captions.

Capture

  1. Set OBS canvas + output resolution to 1280×800.
  2. Video settings: 60 fps capture, 30 fps output, CRF 18-22.
  3. Pre-arrange the screen layout matching Shot 4 (manifest), Shot 5 (SM PDF side-by-side with sm_axiom_table.tex), Shot 6 (notebook running), and Shot 7 (terminal with lake build).
  4. Hit record. Run through the 8 shots matching the timings in the storyboard. Target 2:00 total.
  5. Export MP4 under 50 MB so it fits as a GitHub release asset.

Captioning

  1. Transcribe caption text from the storyboard into a .srt sidecar file.
  2. Upload MP4 and SRT together; GitHub and Zenodo both render SRT captions inline.

Distribution

  • Attach MP4 + SRT to the GitHub Release for v1.0.0-submission-submission.
  • Link the video from README "Quick start" section.
  • If K* Verification requests a supplementary video, attach the MP4 directly.
  • Mirror to Zenodo record when DOI is minted.

Post-submission watchdog

  • Weekly heartbeat workflow enabled (.github/workflows/heartbeat.yml)
  • Issue templates surface in the GitHub New Issue picker
  • GitHub Pages deploy active (https://kootru-repo.github.io/kstar-verify/)
  • Binder link in VERIFICATION.md resolves
  • Dockerfile.lean4 builds cleanly on a fresh runner

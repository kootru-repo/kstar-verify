Recording notes

  • Use a light color scheme (VS Code "Light Modern" or equivalent) so captions remain legible after compression.
  • Record at 60 fps; export at 30 fps to halve file size without losing perceptible smoothness.
  • Keep each shot < 22 s so a reviewer can index to any segment.
  • Avoid voice-over unless you're a native speaker of the target reviewer's language; captions are more accessible.

Human-gated: actually recording the video

The storyboard above is ready to record. Below is the end-to-end recording procedure that must be performed manually (tools cannot record your screen).

Prerequisites

  • OBS Studio (free, cross-platform) installed.
  • A clean browser profile so no bookmarks or extensions distract on camera.
  • A fresh clone of kstar-verify at the submission tag, already built (lake build done).
  • Jupyter server already running with the notebook loaded.
  • A terminal window sized to exactly match the screencast resolution.

Recording workflow

  1. Open OBS Studio. Set canvas + output resolution to 1280×800. Add a Display Capture source for the primary monitor.
  2. In OBS settings → Video: 60 fps capture, 30 fps output.
  3. In OBS settings → Output: MP4 container, CRF 18-22 (higher quality end), x264 encoder.
  4. Arrange scenes for each of the 8 shots (easier to cut between than re-drag sources).
  5. Hit Start Recording, run each shot in the order above, matching the timings. Aim for the 2:00 total. Add a caption overlay track in post.
  6. Export to MP4, target file size < 50 MB so GitHub can host it as a release asset.

Captioning

Use OBS's text source for live captions, OR add them in post with a tool like OpenShot or Kdenlive. Export a .srt sidecar file alongside the MP4; GitHub and Zenodo both render SRT captions inline.

Post-recording

  1. Upload MP4 + SRT as assets on the GitHub Release for tag v1.0.0-submission.
  2. Link the video from the README Quick Start section.
  3. Mirror to Zenodo record when the DOI is minted.
  4. If submission asks for "supplementary video", attach the MP4 directly.

When to re-record

Re-record if:

  • The manifest schema changes (Shot 4 view becomes stale).
  • The SM axiom table layout changes (Shot 5 view becomes stale).
  • The notebook cell count or output changes materially (Shot 6 drifts).
  • Anyone reports a reviewer couldn't reproduce a step.
